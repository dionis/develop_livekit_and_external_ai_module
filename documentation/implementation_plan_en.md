# ARTalk Visual Plugin for LiveKit

This document describes the implementation proposal for a visual plugin in LiveKit to integrate the **ARTalk** model (automatic audio-driven 3D talking head generator).
The design is based on the structure of the `livekit-plugins-ditto` example plugin, adapted to use ARTalk's inference engine.

## Structural Design Proposal

The plugin will be packaged as `livekit-plugins-artalk` and will consist of the following main components:

### 1. `livekit/plugins/artalk/avatar.py`
This file will define the avatar session (`ARTalkAvatarSession`) that connects the LiveKit room with ARTalk.
- **TTSWrapper & TTSWrapperStream**: Override the Text-to-Speech (TTS) behavior defined in `livekit.agents`. Their objective is to send the audio generated by voice synthesis both to the LiveKit room (for the user to hear) and to ARTalk's input stream to generate frames (lipsync).
- **`ARTalkAvatarSession`**: Controls the life cycle of the virtual avatar (startup, connection to the LiveKit Room, and termination) and launches an asynchronous task dedicated to collecting and publishing the frames emitted by ARTalk in video format.

### 2. `livekit/plugins/artalk/artalk_sdk.py`
SDK Wrapper around ARTalk's native implementation, ideally interacting with its internal rendering and Gaussian Splatting inference functions.
- **`ARTalkSDKWrapper`**: Will load the hosted ARTalk model (making use of `torchaudio`, `torch`, and the `diff-gaussian-rasterization` Gaussian rasterizer).
- It will provide methods like `process_audio_chunk(audio_bytes)` to send bursts of floating-point PCM audio and `get_frame_queue()` to consume the rendered *frames* by ARTalk (HxWxC RGB or RGBA arrays).

### 3. `livekit/plugins/artalk/video_source.py`
Video handler that translates numeric arrays into a WebRTC-compatible flow using the LiveKit SDK.
- Uses `rtc.VideoSource` and `rtc.LocalVideoTrack.create_video_track` to create the video track to be published in the `Room`.
- Contains the `publish_frame` loop responsible for changing formats, resizing (if dictated by ARTalk), and calling `capture_frame()`.

## Troubleshooting

### 1. FFmpeg Dependency Errors (`libavfilter.so.9` / `libavutil.so.57`)
ARTalk relies heavily on `torchaudio` for parsing and resampling audio. In isolated Conda environments, recent `torchaudio` versions might require older FFmpeg libraries. 
**Solution**: Ensure `ffmpeg<7` is installed in the conda environment via `conda-forge`, rather than relying solely on the system `apt` implementation, to satisfy these dynamic linker bounds.

### 2. Savitzky-Golay Filter Crash (`window_length must be less than or equal to the size of x`)
When translating TTS audio chunks into motion codes, ARTalk applies a `scipy.signal.savgol_filter` with a `window_length=9` to smooth facial expressions and head poses. If the incoming chunk is smaller than 0.36 seconds (yielding less than 9 frames), the engine crashes.
**Solution**: The SDK uses an internal aggregation buffer (`self._audio_buffer`) to only forward chunks larger than a threshold (e.g., 1.0s) to the inference engine, with a strict minimum cutoff of 0.25s to discard tiny remnants after the stream flushes.

### 3. Rendering Glitches / Diagonal Video Stripes
Because `np.dstack` and `np.transpose` produce transposed memory slices, pushing `frame.tobytes()` natively into LiveKit results in a stride break where pixels shift diagonally and create visual artifacts.
**Solution**: Always guarantee `np.ascontiguousarray(frame)` before piping RGB matrices into the `rtc.VideoFrame` wrapper. It is also good practice to lock color ranges via `np.clip(..., 0, 255)` when casting back from StyleGAN's tensors.

### 4. Severe Audio Dropout & '1 FPS' Video Playback
Two architectural chokepoints can block real-time playback:
1. ARTalk's PyTorch engine is synchronous. If placed directly in the TTS async loop, it blocks the main event thread, starving Cartesia and interrupting speech playback.
2. If the renderer pushes a burst of 25 perfectly generated frames into the WebRTC proxy *instantly*, the proxy discards the overlapping bursts and streams the stream at roughly 1FPS.
**Solution**: 
- Isolate the heavy `process_audio_chunk` bindings inside an `await asyncio.to_thread()` wrapper.
- Pace the WebRTC publication queue loop mathematically using the LiveKit event loop, ensuring frames drain smoothly at exactly `1.0 / target_fps`.
